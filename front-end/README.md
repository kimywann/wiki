# HTTP (HyperText Transfer Protocol)

## HTTP란?

HTTP(HyperText Transfer Protocol)는 **웹에서 클라이언트(브라우저)와 서버가 데이터를 주고받기 위한 프로토콜**입니다.  
기본적으로 **무상태(stateless) 프로토콜**이며, 요청과 응답이 독립적으로 처리됩니다.

<br/>

## HTTP vs HTTPS

| 프로토콜  | 설명                                |
| --------- | ----------------------------------- |
| **HTTP**  | 데이터를 평문으로 전송 (보안 취약)  |
| **HTTPS** | TLS/SSL을 이용한 암호화 (보안 강화) |

- HTTPS를 사용하면 **데이터 암호화, 무결성 보장, 인증서 기반 신뢰성 확보**가 가능합니다.

<br />

# HTTP 메서드

| 메서드   | 설명                        |
| -------- | --------------------------- |
| `GET`    | 리소스를 조회 (캐싱 가능)   |
| `POST`   | 리소스를 생성 (데이터 전송) |
| `PUT`    | 리소스를 수정 (전체 변경)   |
| `PATCH`  | 리소스를 수정 (부분 변경)   |
| `DELETE` | 리소스를 삭제               |

# 주요 HTTP 상태 코드

HTTP 상태 코드는 **클라이언트의 요청에 대한 서버 응답의 상태를 나타내는 숫자 코드**입니다.

## 🔹 1xx (Informational) - 정보 응답

| 코드                      | 의미                                                         |
| ------------------------- | ------------------------------------------------------------ |
| `100 Continue`            | 요청의 일부를 수신했으며, 클라이언트가 계속 요청을 보내도 됨 |
| `101 Switching Protocols` | 프로토콜 변경 요청을 승인함                                  |

## 🔹 2xx (Success) - 성공 응답

| 코드             | 의미                                    |
| ---------------- | --------------------------------------- |
| `200 OK`         | 요청이 성공적으로 처리됨                |
| `201 Created`    | 새로운 리소스가 성공적으로 생성됨       |
| `204 No Content` | 요청이 성공했지만, 반환할 콘텐츠가 없음 |

## 🔹 3xx (Redirection) - 리디렉션

| 코드                    | 의미                                             |
| ----------------------- | ------------------------------------------------ |
| `301 Moved Permanently` | 리소스가 영구적으로 이동됨 (SEO에서 중요)        |
| `302 Found`             | 리소스가 임시적으로 이동됨                       |
| `304 Not Modified`      | 캐시된 리소스를 사용할 수 있음 (네트워크 최적화) |

## 🔹 4xx (Client Error) - 클라이언트 오류

| 코드               | 의미                         |
| ------------------ | ---------------------------- |
| `400 Bad Request`  | 잘못된 요청 (문법 오류 등)   |
| `401 Unauthorized` | 인증이 필요함                |
| `403 Forbidden`    | 접근이 금지됨                |
| `404 Not Found`    | 요청한 리소스를 찾을 수 없음 |

## 🔹 5xx (Server Error) - 서버 오류

| 코드                        | 의미                                    |
| --------------------------- | --------------------------------------- |
| `500 Internal Server Error` | 서버 내부 오류 발생                     |
| `502 Bad Gateway`           | 게이트웨이(프록시)가 잘못된 응답을 받음 |
| `503 Service Unavailable`   | 서버가 일시적으로 사용 불가능           |

# HTTP/1.1 vs HTTP/2 vs HTTP/3

## 🔹 HTTP 버전별 차이점

| 특징               | HTTP/1.1                               | HTTP/2                 | HTTP/3                      |
| ------------------ | -------------------------------------- | ---------------------- | --------------------------- |
| **전송 방식**      | 텍스트 기반                            | 바이너리 기반          | QUIC 기반 (UDP)             |
| **다중 요청 처리** | 직렬 처리 (Head-of-line blocking 발생) | 멀티플렉싱 (동시 처리) | 개선된 멀티플렉싱           |
| **서버 푸시**      | ❌                                     | ✅                     | ✅                          |
| **압축 방식**      | Gzip                                   | HPACK (헤더 압축)      | QPACK (더 개선된 헤더 압축) |
| **연결 방식**      | TCP                                    | TCP                    | UDP + TLS 1.3               |
| **지연시간**       | 상대적으로 김                          | 개선됨                 | 더 빠름                     |

## 🔹 HTTP/2의 주요 특징

1. **멀티플렉싱 (Multiplexing)**

   - 하나의 TCP 연결로 여러 개의 요청/응답을 동시에 처리 가능
   - HTTP/1.1에서는 하나의 요청이 끝나야 다음 요청을 보낼 수 있었음

2. **서버 푸시 (Server Push)**

   - 클라이언트가 요청하지 않은 리소스를 미리 보내줌 → 로딩 속도 향상

3. **헤더 압축 (HPACK)**
   - 중복되는 헤더를 압축하여 데이터 크기를 줄임

<br />

## 🔹 HTTP/3의 주요 특징

1. **QUIC(Quick UDP Internet Connections) 기반**

   - UDP를 사용하여 TCP의 한계를 극복 (3-way handshake 제거)
   - 더 빠른 연결 수립

2. **Head-of-Line Blocking 해결**

   - HTTP/2는 하나의 스트림에서 지연이 발생하면 전체 요청이 지연됨
   - HTTP/3에서는 개별 스트림이 독립적으로 동작하여 지연 없이 처리 가능

3. **TLS 1.3 기본 적용**
   - 보안성을 높이고, 연결 속도를 향상
